## 1. 前提：渠道能力边界确认

在动手改造任何技能之前，必须先完成能力验证。以下是当前已确认、未确认、和确认不存在的边界。

## 2. 妙想日限额与调用策略

### 2.1 日限额明细

| 技能 | 日限额 | 典型单次执行消耗 | 日可支撑任务数 |
|------|--------|-----------------|-------------|
| **mx-data** | **150 次** | 3-5 次（investment-research 深度分析） | 30-50 个研究任务 |
| **mx-search** | **150 次** | 1-2 次（每个研究任务的资讯搜索） | 75-150 个研究任务 |
| **mx-xuangu** | **150 次** | 1 次（每次全市场筛选） | 150 次筛选 |
| mx-zixuan | 200 次 | 1-2 次（查询/添加自选） | 充足 |
| mx-moni | 1000 次 | 1-5 次（买卖/持仓查询） | 极充足 |
| mx-poster | 20 次 | 1-2 次（发帖/互动） | 10-20 次社区操作 |

### 2.2 调用策略：理杏仁承担高频，妙想承担特色场景

```
                    日可调用量对比
mx-data    ████████░░░░░░░░░░░░░░  150
mx-search  ████████░░░░░░░░░░░░░░  150
mx-xuangu  ████████░░░░░░░░░░░░░░  150
理杏仁      ████████████████████████████████████████████████████  144,000/天 (1000/min × 60 × 24)

结论：理杏仁带宽 → 极大，妙想带宽 → 有硬约束
```

**核心策略**：

| 场景 | 策略 |
|------|------|
| 批量/高频/自动化取数 | **走理杏仁**（几乎无限的带宽） |
| 交互式查询、即席探索 | **走 mx-data**（便捷性优先） |
| 金融资讯搜索 | **走 mx-search**（唯一来源，必须走） |
| 全市场条件筛选 | **走 mx-xuangu**（唯一来源，必须走） |
| 关键数据交叉验证 | 理杏仁主取 + mx-data 抽查核对（非每次执行都双重取值） |

**触达限额时的降级**：mx-data/mx-search 超限（code=113）→ 降级到理杏仁（若该维度理杏仁存在）或 WebSearch（若不存在）。

### 2.3 "假设任务量"下的日消耗估算

| 日任务量 | mx-data 消耗 | mx-search 消耗 | mx-xuangu 消耗 | 是否超限？ |
|---------|-------------|---------------|---------------|----------|
| 轻量（1-2 个深度研究） | 5-10 | 2-4 | 0-1 | ✅ 充裕 |
| 中等（5 个深度研究 + 1 次全市场筛选） | 15-25 | 5-10 | 1 | ✅ 充裕 |
| 重度（10 个深度研究 + 3 次筛选 + 2 次组合审视） | 30-50 | 10-20 | 3 | ✅ 充裕 |
| 极端（20+ 个研究任务） | 60-100 | 20-40 | 5 | ⚠️ 接近上限 |

**结论**：在正常的日使用量下，150 次/天的限额足够。仅当进行大规模批量自动化处理（如"对 A 股全市场 5000 只股票逐一分析"）时才会有压力——但这不在本项目的设计用例范围内。

## 3. 技能增强分析（按受益分级）

### 分级标准（修订版）

| 分级 | 定义 | 判定依据 |
|------|------|---------|
| 🔴 革命级 | 当前存在明确的、可量化的大痛点，双渠道直接解决 | 本次 601336 研究验证了痛点真实性 |
| 🟠 质变级 | 当前能完成任务但效率低/质量差，双渠道显著提升质量和效率 | 渠道能替代当前方法的大部分手工步骤 |
| 🟡 增强级 | 当前方法基本可用，双渠道增加精度或便利性 | 锦上添花，非必需 |
| 🟢 微调级 | 数据需求弱，或与数据渠道关系不大 | 文档更新/思维框架类 |

---

### 🔴 革命级受益（3 个技能）

#### 3.1 `/investment-research` — 四大师综合分析

**当前痛点（601336 实证）**：
- 3 个并行 Agent 全部因权限问题失败
- 保险行业报表：东方财富 API 返回"报表配置不存在"
- 偿付能力/代理人数据：完全缺失
- PEV 估值：靠手动计算（仅一个新闻来源）
- 股东结构：API 返回空
- 宏观利率背景：缺失

**增强方案**：

| 数据维度 | 增强前 | 增强后（渠道/方法） | 确定性 |
|---------|--------|-------------------|--------|
| 结构化财报 | WebSearch 拼凑 + 东财API（保险不可用） | 理杏仁 `fs/insurance\|bank\|security` 自动适配 | ✅ 已验证 |
| 估值分位点 | 估算或缺失 | 理杏仁 6×8 分位点矩阵，自动注入报告 | ✅ 已验证 |
| 前十大股东 | API 空返回 → 缺失 | 理杏仁 `majority-shareholders` | ✅ 已验证 |
| 高管增减持 | 完全缺失 | 理杏仁 `senior-executive-shares-change` | ✅ 已验证 |
| 宏观利率 | 缺失 | 理杏仁 `macro/national-debt` 10年期国债收益率 | ✅ 已验证 |
| **保险 EV/NBV** | **仅一篇金融界新闻补漏** | **理杏仁 insurance 报表（⚠️ 需 P4.1 验证 EV/NBV 是否在字段中）** | **⚠️ 待验证** |
| 行业估值对比 | 手工估算同行 PEV | 理杏仁 `industry/fs/sw/insurance` + mx-xuangu 成分股 | ✅ 已验证 |
| 金融资讯 | WebSearch（不稳定） | mx-search（信源权威筛选） | ✅ 已验证 |
| 实时行情快照 | 腾讯 qt（GBK 乱码） | mx-data "最新价 涨跌幅 PE PB" | ✅ 已验证 |

**执行流程变化**：

```
增强前:
  启动 3 个 Agent → WebSearch → 手动整理 → 写报告
  (60%+ Agent 因权限失败, 保险数据完全缺失)

增强后:
  解析目标公司代码
    → 理杏仁批量拉取: 财报 + 估值分位点 + 股东 + 治理 + 宏观 (5-8次API)
    → mx-data 补充: 实时行情快照 (1次)
    → mx-search 补充: 最新重要资讯 (1次)
    → 理杏仁交叉验证: 关键估值数据 vs mx-data 对比 (可选)
    → 结构化数据注入报告模板
    → 4 大师视角只负责"分析"而非"取数"
```

> ⚠️ **EV/NBV 声明修正**：保险专属财报 357 字段可大幅提升数据完整性，但 EV/NBV 是否在其中需 P4.1 验证。若不在，该维度仍标注"需年报原文"。

#### 3.2 `/management-deep-dive` — 管理层纵深研究

**当前痛点**：完全依赖 WebSearch 收集管理层信息，无法量化验证"言行一致"。

**增强方案**：

| 增强点 | 渠道 | 效果 |
|--------|------|------|
| 高管增减持完整历史 | 理杏仁 `senior-executive-shares-change` | **量化"言行一致"**。管理层说看好但减没减持——数据直接回答 |
| 大股东增减持（含均价） | 理杏仁 `major-shareholders-shares-change` | 区分"真增持"和"象征性增持" |
| 股东人数变化趋势 | 理杏仁 `shareholders-num` | 筹码集中还是分散 |
| 管理层访谈/媒体报道 | mx-search | 比通用搜索更精准的财经媒体覆盖 |
| 公司公告（问询函/监管措施） | 理杏仁 `inquiry` + `measures` | 监管关注记录 |

**核心转变**：这个技能从"定性文本分析"变为"定量数据 + 定性文本"双层分析。定量数据提供骨架（增减持记录、筹码趋势），定性文本提供血肉（言论、媒体报道）。

#### 3.3 `/quality-screen` — 去劣筛选

**当前痛点**：对每家公司逐一 WebSearch 取 7 条指标，效率极低。

**增强方案**（修订版——承认 mx-xuangu 筛选范围有限）：

```
Step 1 (mx-xuangu 初步筛选):
  用确认支持的条件快速缩小范围：
    "ROE>15% AND 净利润增速>0 AND PE<30 AND 行业=消费"
  → 返回候选池（通常 50-200 只）

Step 2 (理杏仁批量取值):
  对候选池中每只股票，批量拉取 mx-xuangu 不支持的指标：
    毛利率、净利率、资产负债率、FCF、收入增速
  → 理杏仁支持 batch stockCodes，候选池一次或少量几次 API 可全覆盖

Step 3 (本地最终筛选):
  用完整 7 指标对候选池做最终过滤和排名
  → 输出 Top N
```

> ⚠️ **mx-xuangu 筛选能力修正**：确认支持 PE/PB/ROE/净利润增速/股息率/行业限定/涨跌幅/成交量/股价区间。毛利率/净利率/资产负债率/FCF/收入增速的支持状态待验证。未支持的指标走 Step 2 理杏仁批量取值。

### 🟠 质变级受益（6 个技能）

#### 3.4 `/news-pulse` — 公司新闻脉搏

**当前**：4 个 Agent 并行 WebSearch（公司事件/监管/行业/情绪）。

**增强方案（修订版——不替代多 Agent 架构）**：

```
增强后架构：
  4 个 Agent 的并行架构保留（不同角度的搜索仍有价值）
    Agent 1 (公司事件): WebSearch → mx-search "东方财富最新公告"
    Agent 2 (监管政策): WebSearch → mx-search "新能源汽车产业政策最新解读"
    Agent 3 (行业动态): WebSearch → mx-search "白酒板块近期新闻"
    Agent 4 (市场情绪): 理杏仁 margin-trading + 陆股通 + mx-data 资金流向

关键变化：每个 Agent 内部从 WebSearch 改为 mx-search，而非取消 Agent 并行架构
```

**增强效果**：
- 信源质量：通用搜索引擎 → 金融信源智能筛选
- 事件检测：新增理杏仁龙虎榜/大宗交易/高管增减持作为**自动事件触发器**
- Token 消耗：每个 Agent 减少约 60%（不再需要从 HTML 中提取信息）

> ⚠️ **修正**：原来的"4 Agent → 1 mx-search"声称过于简化。mx-search 不能替代多角度并行搜索的架构价值。

## 4. 跨技能系统性增强

### 4.1 预加载策略：从"逐项取数"到"批量数据包"

当前模式：每个分析模块独立取数 → 多次 API 调用 → 重复拉取相同数据。

增强模式：

```
Skill 启动
  → 解析目标公司列表 + 所需数据维度
  → 一次性批量拉取"数据包"：
      理杏仁: 财报(5年) + 估值(当前+分位) + 股东 + 治理 (3-5 次 API, batch stockCodes)
      mx-data: 实时行情快照 (1 次)
      mx-search: 最新重要资讯 (1 次)
  → 缓存数据包（跨模块共享 TTL 1 小时）
  → 各分析模块从共享数据包读取，专注分析逻辑
```

**收益**：
- API 调用从 15-20 次压缩为 5-7 次
- MX 日限额压力大幅降低
- 各分析模块代码简化（不关心数据从哪来）

### 4.2 估值分析四维坐标

从"PE 是多少 → 和同行比"升级为：

```
维度 1 (理杏仁): 当前 PE 处于自身历史的什么分位？
维度 2 (理杏仁): 当前 PE 处于申万同行业的什么分位？
维度 3 (mx-xuangu): 全市场有多少公司比我便宜且质量不低于我？
维度 4 (理杏仁 macro): 在当前利率环境下，这个 PE 是高是低？
```

用于所有涉及估值判断的技能（investment-research, industry-funnel, portfolio-review, thesis-tracker）。

### 4.3 交叉验证策略（修订版——承认口径差异）

```
适用场景: 仅在以下条件同时满足时执行交叉验证
  1. 理杏仁和 mx-data 返回的同一指标
  2. 该指标的定义在两个来源中相同（如都是 TTM PE）
  3. 数值差异 > 2%（放宽阈值，原来 1% 对 NLP 源太严格）

不适用场景:
  - NLP 提取的数值 vs 结构化编码字段（口径可能不同，不具可比性）
  - mx-search 返回的定性信息
  - 理杏仁独有的分位点数据

交叉验证方式:
  - 差异 < 2%: 静默通过，标注两个来源
  - 差异 2-5%: 报告注明"双源存在轻微差异"，优先采纳理杏仁
  - 差异 > 5%: 标记为"数据异常"，建议人工核查年报
```

### 4.4 MX 技能调用模式优化

当前所有 MX 技能输出到文件再读取。高频场景优化：

| 场景 | 优化策略 |
|------|---------|
| investment-research | 理杏仁承担 80% 取数，MX 仅承担 tick 行情 + 资讯搜索 |
| quality-screen | mx-xuangu 初筛（1 次）+ 理杏仁批量取值 |
| portfolio-review | 理杏仁 batch API 批量拉取持仓，mx-search 逐个搜索新闻 |
| 数据交叉验证 | 仅在报告出稿前执行一次，非常规操作 |
